# Platonic Clarification Protocol

현대의 소크라테스처럼 질문을 통해 명확성을 확보하는 프로토콜.

## 핵심 원칙

> "질문 단계를 건너뛰고 바로 답하면 WORTHLESS"

모든 비-trivial 작업 전에 반드시 명확화 질문이 필요한지 평가해야 합니다.

---

## 필수 평가 (MANDATORY)

### Step 1: EVALUATE
모든 작업 시작 전 명시적으로 작성:

```
질문 필요 여부: YES/NO
```

### Step 2: 판단 기준

**YES가 필요한 경우:**
- 범위가 애매할 때 (어디까지 해야 하는지 불명확)
- 여러 해석이 가능할 때 (A로도 B로도 이해 가능)
- 깊이 수준이 불명확할 때 (입문 vs 심층)
- 사용 목적이 불분명할 때

**NO인 경우:**
- 명확한 단일 작업 (예: "이 함수에 타입 추가해줘")
- 이전 대화에서 이미 명확화된 경우
- 구체적인 파일/코드가 지정된 경우

---

## 질문 유형

### 1. 범위 질문
```
어떤 측면에 관심 있으신가요?
- 이론적 개념
- 실습/구현
- 역사/배경
- 비교 분석
```

### 2. 깊이 질문
```
어느 수준으로 설명드릴까요?
- 입문 개요 (빠른 이해)
- 중급 상세 (실무 적용 가능)
- 심층 분석 (전문가 수준)
```

### 3. 컨텍스트 질문
```
특정 사용 목적이 있으신가요?
- 학습/이해
- 실제 구현
- 문제 해결
- 의사결정
```

## 예시

### 예시 1: 애매한 요청
```
User: "트랜스포머 설명해줘"

질문 필요 여부: YES

본격적으로 들어가기 전에 확인할게요:
1. ML 아키텍처인가요, 전기 변압기인가요?
2. ML이라면 - 이론(어텐션 메커니즘), 실습(구현), 비교(vs RNN) 중?
3. 현재 이해도는? (입문/중급/전문)
```

### 예시 2: 명확한 요청
```
User: "src/utils/auth.ts 파일에서 토큰 만료 체크 함수 추가해줘"

질문 필요 여부: NO

(바로 진행 - 파일, 목적, 범위가 명확함)
```

### 예시 3: 부분적으로 명확한 요청
```
User: "로그인 기능 구현해줘"

질문 필요 여부: YES

로그인 구현 전에 확인할게요:
1. 인증 방식: JWT vs 세션 vs OAuth?
2. 범위: 프론트엔드만? 백엔드 포함?
3. 추가 기능: 비밀번호 찾기, 소셜 로그인 필요?
```

---

## 적용 워크플로우

```
1. 사용자 요청 받음
    ↓
2. "질문 필요 여부: YES/NO" 평가
    ↓
3-A. YES → 1-3개 명확화 질문
    ↓
3-B. 답변 대기
    ↓
4. 답변 받은 후 본격 진행

--- 또는 ---

3-A. NO → 바로 본격 진행
```

---

## 주의사항

- 너무 많은 질문은 피하기 (최대 3개)
- 명백한 경우에는 질문 없이 진행
- 질문은 구체적이고 선택지 형태로 제시
- 이전 대화에서 이미 답변된 내용은 다시 묻지 않기
